# veo-tools 2.0.0 — "Foundation"

**Veo Foundation** is the enabling release for the Veo improvements roadmap (sub-project 1 of 5). It extracts a shared `@veo-core/*` library, adds the cross-cutting Veo API parameters that were previously missing, makes the audio default context-aware, and centralizes configuration validation so bad requests fail *before* a paid API call.

Targets **Vertex AI** (`predictLongRunning`). Verified live against the API on 2026-06-17.

---

## ⚠️ Breaking changes

This is a major release. Two changes can affect existing callers:

1. **`generateVideo()` signature** — the output destination moved into the config object.
2. **Audio default** — was always *off*, now *context-aware* (and on by default for most use cases).

See **Migrating from 1.x** below for exact before/after.

---

## 🚀 Migrating from 1.x

### 1. `generateVideo()` — output destination is now in the config

The second positional argument is gone. Pass `outputPath` (local file) **or** `storageUri` (GCS server-side delivery) inside the config — exactly one is required (enforced by validation rule #9).

```diff
  // Local download
- await generateVideo(config, '/path/out.mp4')
+ await generateVideo({ ...config, outputPath: '/path/out.mp4' })

  // Server-side delivery to GCS (no local download)
+ await generateVideo({ ...config, storageUri: 'gs://my-bucket/out/' })
```

If you set neither, or both, you get a clear validation error rather than a late failure.

**CLI users are unaffected** — `veo-generate.ts --output …` / `--storage-uri …` already worked this way; only the programmatic library signature changed.

### 2. Audio is now context-aware

In 1.x audio was always off unless you opted in. In 2.0 the default is derived from the Phase 1 USE CASE:

| Use case | Audio default |
|---|---|
| `social`, `marketing`, `product`, `storytelling` | **on** |
| `hero-background`, `ambient`, `loop` | off |
| unspecified | **on** (matches the Veo 3.1 API native default) |

Audio generation costs more than silent video. To keep the previous (silent) behavior:

```bash
# CLI
veo-generate.ts --prompt "…" --output out.mp4 --no-audio
```
```diff
  // Library
+ await generateVideo({ ...config, outputPath: 'out.mp4', generateAudio: false })
```

Phase 4 PRESENT always shows the resolved audio state (with the reason) before generation, so you can catch an unexpected `audio: on` before spending.

### Migration checklist

- [ ] Replace every `generateVideo(cfg, path)` call with `generateVideo({ ...cfg, outputPath: path })`.
- [ ] Decide audio per use case; add `--no-audio` / `generateAudio: false` where you want the old silent default.
- [ ] If you relied on `gcloud` being on PATH for auth, note that auth now uses `google-auth-library` (ADC / service account / Workload Identity) — no `gcloud` shell-out.
- [ ] If you set `GOOGLE_CLOUD_PROJECT_ID`, it still works (fallback preserved), but `GOOGLE_CLOUD_PROJECT` is preferred.

## ✨ New features

- **Cross-cutting parameters / CLI flags**: `--negative-prompt`, `--seed`, `--sample-count`, `--person-generation` (`allow_all` / `allow_adult` / `disallow`), `--storage-uri` (GCS delivery), `--add-watermark` / `--no-add-watermark` (SynthID), `--include-rai-reason`, and **4K** resolution.
- **Config validation** — `validateConfig()` catches model/duration/resolution conflicts, Veo 2 audio + resolution limits, per-model max resolution (Lite 1080p), `sampleCount` range/integer check, regional `personGeneration` restrictions, `seed` range, `personGeneration` and `resolution` enums, and the Veo 3 "enhancement always on" constraint. Returns auto-fixes + suggestions; **never throws**.
- **Cost estimation** — `estimateCost()` keyed by model × resolution × duration × audio × sampleCount, surfaced by a `--dry-run` PRESENT preview (settings + cost, no API call).
- **Audio lexicon** reference and audio-prompt examples for the 6-phase workflow.

## 🔧 Under the hood

- Shared `skills/_shared/veo-core/` library (auth, api, validation, pricing, generate, image-helpers). `veo-generate.ts` (594→55 lines) and `veo-multi-generate.ts` (635→33) are now thin entry points over `@veo-core/*`.
- Auth via `google-auth-library` instead of a `gcloud` shell-out — works in CI and containers.
- Hardened downloads: cross-origin Authorization stripping (RFC 6454), HTTPS→HTTP redirect rejection, redirect cap, atomic temp-file writes, dual timeouts (incl. pre-header watchdog), and a transient-only poll retry (survives "high load", re-throws permanent errors immediately).
- CI workflow: `tsc --noEmit` typecheck + the unit-test suite on every PR.

## ✅ Verified live (Vertex AI, project probe 2026-06-17)

- All 6 `AVAILABLE_MODELS` respond and generate valid MP4s.
- Default delivery returns the video **inline as base64** (no `storageUri`) — handled correctly.
- Lite `sampleCount=4` confirmed; `enhancePrompt=false` is rejected by Veo 3 (caught client-side by rule #11 before the paid call).

## 📋 Known limitations

- **Pricing figures are estimates.** The probe verified model liveness and the response shape, not exact per-second prices — confirm against the official Vertex AI Veo pricing page before relying on cost output for billing.
- `sampleCount > 1` generates (and bills) N videos but Foundation retrieves only the first; a warning is surfaced in `GenerationResult.warnings`.
- The Veo 3.0 family is deprecated (discontinuation 30 Jun 2026) and kept only for backwards-compat; re-run the model probe if deploying after that date.
